# RAG Pipeline Implementation Guide

This document provides a detailed breakdown of the RAG (Retrieval Augmented Generation) pipeline implementation for the ERP RBAC-RAG system.

## 1. Pipeline Overview

The RAG pipeline is responsible for taking a user's query, retrieving relevant information from the knowledge base, and generating a response using a Large Language Model (LLM), all while enforcing the user's access rights.

The pipeline consists of the following steps:

1.  **Querying**
2.  **Security Filtering (Pre-retrieval)**
3.  **Document Retrieval**
4.  **Context Augmentation**
5.  **LLM Prompting**
6.  **Response Generation**
7.  **Security Filtering (Post-generation)**
8.  **Response to User**

## 2. Step-by-Step Implementation

### 2.1. Querying

-   The user submits a query through the frontend application.
-   The frontend sends the query to the backend's RAG API endpoint.

### 2.2. Security Filtering (Pre-retrieval)

-   Before any data is retrieved, the system must check if the user has the necessary permissions.
-   The user's profile (including their role and attributes) is sent to Cerbos along with the query.
-   Cerbos policies determine what kind of information the user is allowed to access. For example, a policy might restrict access to documents belonging to a specific department.

### 2.3. Document Retrieval

-   **Query Embedding**: The user's query is converted into a vector embedding using a sentence transformer model.
-   **Vector Search**: The query embedding is used to search the vector database (e.g., ChromaDB) for the most similar document chunks.
-   **Access Control Filtering**: The search results from the vector database are filtered based on the user's access rights, as determined by Cerbos in the previous step. This ensures that only documents the user is allowed to see are considered for the next step.

### 2.4. Context Augmentation

-   The content of the retrieved document chunks is combined to form a context.
-   This context will be used to augment the LLM's knowledge.

### 2.5. LLM Prompting

-   A prompt is constructed using the original user query and the retrieved context.
-   The prompt should be carefully designed to instruct the LLM to answer the query based on the provided context.

### 2.6. Response Generation

-   The prompt is sent to the LLM (e.g., OpenAI GPT-4) to generate a response.

### 2.7. Security Filtering (Post-generation)

-   The response generated by the LLM is scanned for any sensitive information that the user is not authorized to see.
-   This is a crucial step to prevent accidental data leakage. For example, the LLM might inadvertently include information from a document that the user has access to, but in a way that reveals information about other documents they don't.

### 2.8. Response to User

-   The final, sanitized response is sent back to the frontend and displayed to the user.

## 3. Implementation in Code

Here's a high-level overview of how this could be implemented in the backend:

# app/rag/services.py

from app.auth.cerbos_client import check_permission
from app.vector_db.service import VectorDBService
from app.llm.service import LLMService

class RAGPipelineService:
    @staticmethod
    async def process_query(user: dict, query: str) -> str:
        # 1. & 2. Security Filtering (Pre-retrieval)
        # This is a simplified example. In a real implementation, you would
        # have more complex logic to determine the resource and action.
        allowed = await check_permission(user, resource="documents", action="read")
        if not allowed:
            return "You do not have permission to perform this action."

        # 3. Document Retrieval
        retrieved_chunks = await VectorDBService.search(query, user['roles'])

        # 4. Context Augmentation
        context = "\n".join([chunk.content for chunk in retrieved_chunks])

        # 5. LLM Prompting
        prompt = f"Context:\n{context}\n\nQuery: {query}\n\nAnswer:"

        # 6. Response Generation
        response = await LLMService.generate_response(prompt)

        # 7. Security Filtering (Post-generation)
        sanitized_response = await self.sanitize_response(response, user)

        # 8. Response to User
        return sanitized_response

    @staticmethod

        return response
